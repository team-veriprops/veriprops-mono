## Executive Summary

Veriprops is a property-verification platform for Nigerians in the Diaspora. The product packages an opinionated, role-scoped verification workflow (Field, Surveyor, Registry, Lawyer agents) into three tiers (Basic / Standard / Premium), produces a versioned **Report** anchored on a public-safe **Verification ID** (`VP-YYYY-XXXXXX`) and a composite 0–100 **Trust Score**.

The PRD sequences 19 phases (Phase 0 → 19) plus a Post-MVP roadmap. Phases 0–10 form the **MVP cut line**. Phases 11–19 harden, scale, and add growth/compliance maturity.

**Stack already chosen:** FastAPI + async SQLAlchemy + Alembic + MySQL on the backend, Next.js 16 App Router + React 19 + Tailwind v4 on the frontend. Cross-app contract is `/api/users/auth/...` shape, camelCase JSON, JWT in HttpOnly cookies.

---

## Goals

1. Make "verified property" the default belief in Nigerian real estate. End state: *"If it's not verified, nobody buys it."*
2. Ship a deterministic, audit-logged, state-machine-driven verification workflow that a non-technical admin can operate.
3. Deliver a defensible **Report** product (PDF + HTML, public lookup summary, share modes) with a clear legal stance: *"We reduce uncertainty. We do not eliminate it."*
4. Build a self-sustaining agent network with KYC, role-specific submission UIs, transparent earnings, and reputation.
5. Reach the MVP cut line (Phase 0–10) with end-to-end demoable vertical slices.

## Non-Goals (per PRD §Post-MVP)

- Property Identity Layer (canonical property entity across multiple verifications).
- Escrow / Transaction Layer (facilitating actual purchase fund flow).
- Auto-assignment AI (initial assignment is admin-driven).
- WhatsApp OTP delivery (SMS is sufficient for MVP).
- Native mobile apps.
- Verification Academy / long-form education content.

## High-Risk Areas

| Risk Area | Why it's risky | Mitigation locus |
|---|---|---|
| **Versioned consent + legal disclaimers** | Veriprops sells *opinions*, not guarantees. Mis-recorded consent = liability exposure. | Phase 0 + every payment/report-view checkpoint |
| **Agent identity leakage** | Customer ↔ agent direct contact undermines the platform's value. API must enforce — not the UI. | Phase 7, Phase 9, Phase 11 — API-layer filtering |
| **Message fraud detection** | "Pay me directly outside the platform" attacks. | Phase 11 send-time scan |
| **State-machine determinism** | Global state must be derived from task states, not set by humans. | Phase 0 (state machine primitive) + Phases 6–8 |
| **OAuth email-collision** | Silent merging = account takeover risk. | Phase 2 §2.2 — explicit reject |
| **Wire transfer reconciliation** | Multi-currency manual reconciliation invites errors. | Phase 5 (initial) + Phase 18 (panel) |
| **PDF/HTML report parity** | Legal disclaimer must appear on every page of both. | Phase 10 |
| **Refund + dispute outcomes** | Money movement on contested judgements. | Phase 14 |
| **Public lookup `noindex` rules** | SEO leak of disputed/in-progress verifications. | Phase 13 |
| **Audit retention / NDPR (Nigerian PII law)** | Indefinite audit retention vs. PII erasure obligations. | Phase 19 |

## Cross-Cutting Concerns

### Auth
- JWT in HttpOnly cookie (15 min access / 30 day refresh).
- Two orthogonal role fields: `user_type` (`USER`/`ADMIN`, immutable) and `user_persona` (multi-valued: `CUSTOMER`, `AGENT`).
- Portal routing: Admin-only → `/admin`; Agent+anything → `/agent`; Customer-only → `/portal`. Highest privilege wins.
- Trust elevation: Customer trusted on first successful payment; Agent trusted on the approval of first submitted task.
- Centralised route protection in Next.js `proxy.ts` — cookie-presence checks only; backend enforces validity per request.

### Data model
- All entities extend `BaseEntity`. Soft delete only.
- **No DB foreign keys, no cascade, no ON DELETE** — application-enforced references via indexed columns.
- Don't create duplicate indexes, prefer UniqueConstraint to create_index.
- camelCase serialisation via `to_camel` alias generator; Python stays snake_case.
- Versioned consent records carry `(user_id, document_type, version, accepted_at, ip, device_fingerprint)`.
- Audit log is append-only, indefinite retention (Phase 19 export).

### Observability
- Structured request logs via `RequestLoggingMiddleware`.
- Audit log on every state transition (Phase 0 hook).
- Real-time dashboard updates: WebSocket *or* SSE — **decision pending** (Open Q22).

### State machines
- **Verification** (global) — derived from task states; only forward-only transitions.
- **Task** (per role) — `PENDING → ASSIGNED → ACCEPTED → IN_PROGRESS → SUBMITTED → APPROVED`. Detours: `ASSIGNED → PENDING`, `SUBMITTED → REJECTED → IN_PROGRESS`.
- **Report** versioning: `v1.0` / `v1.1` / `v2.0` / `v3.0`. Old versions watermarked SUPERSEDED.

## System Implications

### Database
- MySQL with async drivers; serverless-aware engine (NullPool when `DEPLOYMENT_IS_SERVERLESS=true`).
- Tables anticipated post-Phase 0: `users`, `user_consents`, `consent_documents`, `audit_logs`, `signup_drafts`, `oauth_identities`, `sessions`, `otp_attempts`, `agent_applications`, `kyc_records`, `admin_invitations`, `admin_roles`, `verifications`, `properties`, `tasks`, `reports`, `report_versions`, `evidence_items`, `messages`, `payments`, `wire_proofs`, `payouts`, `commission_rules`, `pricing_configs`, `referrals`, `share_links`, `disputes`, `consent_snapshots`.
- Indexing strategy: by `id`, by reference-IDs (e.g. `verification_id`, `user_id`), by status + sort columns for list views.
- Migrations: Alembic autogenerate; never emit `ADD FOREIGN KEY`.

### API
- Mount under `/api`; webhooks under `WEBHOOK_PATH` (default `/webhooks`).
- Auth controller is the contract source for `/api/users/auth/...`.
- Customer-facing endpoints **filter agent identity** to `{role, first_name, avatar_url, verified}` only — enforced at API serialisation.
- Real-time channel: SSE (live dashboard updates, notifications, metrics counters, audit feed), WS (collaborative workflow,chat,presence,two-way realtime commands).

### Security
- HttpOnly + Secure + SameSite cookies; frontend never reads tokens.
- OAuth: signed state, PKCE S256, strict redirect URI, Apple ID-token JWKS validation, no silent merge.
- Login throttle: warn at 5, lockout at 7 (15 min).
- OTP: max 3 resends → 30-min lockout.
- Send-time message fraud scan (phone, email, URL, banking, "outside the platform" phrases).
- Versioned consent enforced at every action that creates legal exposure.
- Encrypted KYC blob storage in S3 with per-user access control.
